# apps/api/drizzle

Database migrations for the API, generated by [drizzle-kit](https://orm.drizzle.team).

## Why it exists

The API uses Drizzle ORM against Postgres. Every schema change is captured as a numbered SQL migration so any environment (local, CI, prod) reaches the same state by replaying them in order. Never hand-edit a migration after it has been committed — write a new one instead.

## Files

- `apps/api/drizzle.config.ts` — drizzle-kit config (reads `DATABASE_URL` from env).
- `apps/api/src/db/schema.ts` — TypeScript source of truth for tables, columns, enums, indexes.
- `apps/api/drizzle/NNNN_*.sql` — generated migrations, applied in filename order.
- `apps/api/drizzle/meta/` — snapshots drizzle-kit uses to diff against `schema.ts`. Do not edit by hand.

## Commands

Run from the repo root.

| Command | What it does | When to run |
|---|---|---|
| `pnpm --filter @webapp/api db:generate` | Diffs `schema.ts` against the latest snapshot and writes a new `NNNN_*.sql` + snapshot. Idempotent when the schema is unchanged. | After editing `schema.ts`. Commit the generated files together with the schema change. |
| `pnpm --filter @webapp/api db:migrate`  | Applies pending migrations against `DATABASE_URL`. | After pulling new migrations or on first local setup. Requires a reachable Postgres (see issue #24). |
| `pnpm --filter @webapp/api db:studio`   | Launches drizzle-studio, a local DB browser. | Ad-hoc inspection. |
| `pnpm --filter @webapp/api db:push`     | Syncs schema directly to the DB without generating a migration. | Avoid in shared branches; dev-only shortcut. |

## Rules for agents

- Never hand-edit a committed migration. Change `schema.ts` and run `db:generate`.
- Commit `drizzle/NNNN_*.sql` **and** the matching `meta/NNNN_snapshot.json` + updated `_journal.json` in the same commit as the schema change.
- `DATABASE_URL` is never committed. Read it from env only.
- Migrations must apply cleanly on an empty database. If you need to backfill data, generate the migration, then add a separate data-migration step — do not mix the two in one file.
- Do not delete or renumber existing migration files. Roll forward, never rewrite history.
